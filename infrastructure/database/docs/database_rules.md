# Database Design Rules

This document defines strict database design rules for the PostgreSQL + PostGIS geospatial ingestion system. These rules currently apply only to the `ref`, `system`, `raw`, and `staging` schemas.

## Design Priorities

- Prioritize referential integrity and auditability over convenience.
- Preserve source lineage from ingestion through normalization.
- Keep naming, key strategy, and data types consistent across schemas.
- Avoid automation that obscures write paths unless it is clearly necessary.

## Naming Conventions

- Use `snake_case` for schemas, tables, columns, constraints, indexes, and SQL identifiers.
- Use plural names for all tables.
- Name foreign key columns as `<target_entity>_id`.
- Name constraints explicitly in SQL, especially foreign keys.
- Prefer clear and stable names over abbreviated or overloaded identifiers.

## Keys And Identity

- Every internal primary key must use `bigint`.
- Generate internal IDs by default using one consistent PostgreSQL pattern: `generated by default as identity` or `bigserial`. Do not mix strategies arbitrarily within the same system.
- Primary keys should be single-column surrogate keys unless there is a strong documented reason otherwise.
- Natural keys from upstream systems may be stored, but they must not replace internal primary keys.

## Foreign Keys And Integrity

- Every logical relationship must be enforced with a foreign key unless there is a documented exception.
- All foreign key constraints must be explicitly named in SQL.
- Foreign key columns must use the exact compatible type of the referenced key.
- Avoid nullable foreign keys unless the relationship is genuinely optional.
- Do not defer referential integrity for convenience during ingestion design.

## Required Data Types

- Use `timestamptz` for all timestamps.
- Use `jsonb` for JSON payload fields.
- Geometry columns must declare an explicit PostGIS geometry type and must always use `SRID 4326`.
- Do not use untyped `geometry` columns.
- Use reference tables instead of database enums.

## Audit Columns

- Every table must include `created_at` where row creation should be recorded.
- Use `updated_at` only for tables whose rows are expected to change after insertion.
- `created_at` and `updated_at`, when present, must be `timestamptz`.
- Do not add triggers for audit timestamps yet unless they are clearly necessary and justified.

## Schema Responsibilities

### `ref`

- Stores controlled reference data used instead of enums.
- Reference values must be stable, documented, and safe to join from operational tables.

### `system`

- Stores ingestion control, review workflow, publish workflow, and audit-oriented operational metadata.
- The source registry belongs here and defines each external source.
- The import batch belongs here and defines one ingestion run.
- The source snapshot belongs here and defines one captured dataset version within a batch.
- Review and publish workflow belongs in the `system` schema.

### `raw`

- Stores untouched imported records from external datasets.
- Raw tables must preserve imported geometry and source tags without normalization.
- Raw records should remain as close as practical to the source payload while still honoring platform-wide type rules.

### `staging`

- Stores normalized candidate records derived from `raw`.
- Staging is the first layer where source data is reshaped for review, matching, and downstream publishing.
- Staging rows must always trace back to `source_snapshot_id`.

## Lineage Rules

- Source lineage must always be traceable from `staging` back to `system_source_snapshots`.
- Every staging table must carry a `source_snapshot_id` foreign key unless a stricter lineage model is formally approved.
- Where staging rows are derived from raw rows, preserve the raw-to-staging relationship explicitly.
- Do not create staging records that cannot be tied to a specific captured dataset version.
- Lineage columns and constraints are mandatory design elements, not optional metadata.

## Workflow Rules

- The source registry defines the external source.
- The import batch defines one ingestion run.
- The source snapshot defines one captured dataset version within a batch.
- Raw tables store untouched imported geometry and tags.
- Staging tables store normalized candidates derived from raw.
- Staging rows must always trace back to `source_snapshot_id`.
- Review and publish workflow belongs in the `system` schema.

## Implementation Rules

- Prefer explicit DDL over implicit behavior.
- Do not create triggers yet unless they are necessary to enforce a rule that cannot be implemented clearly another way.
- Do not introduce enums, polymorphic foreign keys, or convenience shortcuts that weaken auditability.
- Favor append-oriented ingestion design in `raw` and controlled mutation only where the workflow requires it.
- Document any exception to these rules in the relevant migration or schema design note before implementation.
