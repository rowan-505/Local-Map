# Staging To Core Mapping

This document defines the exact business mapping from `staging` to `core` for the geospatial map platform. It describes how reviewed candidate records are promoted into approved, product-ready core records.

## General Promotion Rules

- Promotion is allowed only after review and approval.
- `staging` is an input workflow layer; `core` is the long-term source of truth.
- Promotion must not create permanent foreign key dependencies from `core` back to `staging`.
- Every promoted record must preserve source lineage through core source tables, version rows, or both.
- A promotion action should also be traceable through `system` review and publish workflow records.
- If a reviewed candidate matches an existing core entity, promotion updates that entity instead of inserting a duplicate.
- If no acceptable match exists, promotion inserts a new core entity.

## Cross-Cutting Rules

### Source Lineage

- Preserve lineage from the promoted staging candidate into `core` using source tables and version snapshot payloads.
- The original staging identifiers, source snapshot context, and source references should be captured in `core` lineage records, typically in `core_*_sources` where available and in version snapshot data.
- Promotion should retain enough metadata to explain which reviewed candidate produced the core record and from which source snapshot it came.

### Publish Status

- Initial publish status is assigned by workflow outcome, not by raw import confidence alone.
- For first promotion of an approved entity, use the appropriate `ref.ref_publish_statuses` value based on the approval decision.
- Common initial states are:
  - `approved` when the entity is accepted into core but not yet publicly published.
  - `published` only when the publish workflow actually makes it public.
- Review approval and publish approval are related but separate decisions.

### Initial Version Rows

- Promotion of a place must create an initial row in `core.core_place_versions`.
- The first version row should capture the promoted state as `snapshot_data`, with `version_no = 1` for a new core place.
- If promotion updates an existing place, create the next sequential version row instead of overwriting history.
- The place row should then point `current_version_id` to the newly created version row.

### Review And Publish Workflow

- `system.system_review_tasks` govern review and approval before promotion.
- Promotion should only consume candidates that have passed the intended review path.
- After promotion, a publish workflow record should be created or updated through `system.system_publish_batches` and `system.system_publish_items`.
- Promotion and publishing are linked workflow stages, not the same operation.

### Duplicate Handling

- Duplicate handling is a review decision first and a database write decision second.
- If a staging candidate is determined to represent an existing core entity, merge by updating the matched core record rather than inserting a duplicate.
- If a candidate is a true duplicate of another staging candidate, resolve that in review before core promotion.
- Do not create parallel core entities for the same real-world feature unless there is a clear business reason.

### Admin Area Linkage

- Assign `admin_area_id` in core whenever a reliable reviewed admin area match is available.
- Prefer explicit reviewed admin area linkage over spatial guesswork.
- If reviewed linkage is unavailable but a clearly correct administrative area can be derived safely, assign it during promotion.
- If administrative assignment is still uncertain, leave the nullable link empty and resolve later through review or backfill.

### Lat/Lng Rule

- `core.core_places.lat` and `core.core_places.lng` must be derived from `point_geom`.
- Do not enter `lat` and `lng` independently during normal promotion when `point_geom` is available.
- Independent entry is allowed only for exceptional workflows where geometry must be reconstructed, corrected, or staged separately.

## 1. `staging.staging_admin_area_candidates` -> `core.core_admin_areas` + `core.core_admin_area_names`

### Direct Copies

- `canonical_name` -> `core.core_admin_areas.canonical_name`
- `geom` -> `core.core_admin_areas.geom`
- `centroid` -> `core.core_admin_areas.centroid`
- `admin_level_id` -> `core.core_admin_areas.admin_level_id`

### Normalized Or Transformed

- `matched_core_admin_area_id` determines whether promotion inserts or updates.
- `source_snapshot_id` is not copied directly into the core entity table; it is preserved as lineage metadata in source/version context outside the main admin area row.
- A stable `slug` must be derived from the approved canonical name according to application slug rules.
- `public_id` is generated by core.

### Derived Values

- `source_type_id` is derived from the reviewed candidate lineage and source snapshot context.
- `is_active` defaults to active unless review decides otherwise.
- `created_at` and `updated_at` are generated by core.

### Names

- The approved canonical name should produce a primary row in `core.core_admin_area_names`.
- Additional reviewed names, if available from normalized candidate payloads, should be added as secondary or localized name rows.
- `is_primary` should be true for the canonical display name row.

### Lineage And Promotion Notes

- Promotion should retain the original candidate external ID and source snapshot context in lineage records.
- Parent linkage should be assigned using the reviewed parent candidate if that parent has already been promoted or is promoted in the same workflow batch.

## 2. `staging.staging_place_candidates` + `staging.staging_place_name_candidates` -> `core.core_places` + `core.core_place_names` + `core.core_place_sources` + `core.core_place_versions`

### Direct Copies

- `canonical_name` -> `core.core_places.primary_name`
- approved display value -> `core.core_places.display_name`
- `poi_category_id` -> `core.core_places.category_id`
- `admin_area_candidate_id` -> `core.core_places.admin_area_id` after resolving the referenced core admin area
- `point_geom` -> `core.core_places.point_geom`
- reviewed `matched_core_place_id` determines insert versus update

### Normalized Or Transformed

- `source_entity_type`, `external_id`, and `source_refs` belong in lineage records, not as direct columns on `core.core_places`.
- `name_local` and `secondary_name` should be derived only when reviewed name data clearly supports them.
- `entry_geom` and `footprint_geom` may be populated only if approved geometry exists in normalized candidate data or linked reviewed source content.
- `lat` and `lng` are derived from `point_geom`.
- `confidence_score` may inform review but should not automatically become the final core confidence without product rules; if retained, copy only after approval.

### Derived Values

- `public_id` is generated by core.
- `source_type_id` is derived from candidate lineage and source snapshot context.
- `publish_status_id` is derived from workflow approval and publishing state.
- `importance_score` and `popularity_score` should be initialized from business defaults unless a reviewed scoring process exists.
- `is_public` and `is_verified` are assigned from approval outcomes, not directly from staging.

### Names

- Promote reviewed rows from `staging.staging_place_name_candidates` into `core.core_place_names`.
- The approved primary name should align with `core.core_places.primary_name`.
- Additional reviewed names should become multilingual or alternate rows with retained language, script, and name type metadata.

### Source Lineage

- Create a row in `core.core_place_sources` for the approved source lineage.
- Preserve source type, external ID, source name, captured timestamp if known, and reviewed raw lineage payload where useful.

### Initial Version Row

- Insert a row into `core.core_place_versions` containing the approved place snapshot in `snapshot_data`.
- Set `version_no = 1` for a new place or increment for an update.
- Set `core.core_places.current_version_id` to the created version row.

### Review, Publish, And Duplicates

- Promotion should happen only after place review tasks are resolved appropriately.
- A publish item should be created after the place is inserted or updated in core.
- If the candidate represents an existing place, merge into that core record and create a new version row instead of inserting a duplicate.

## 3. `staging.staging_road_candidates` -> `core.core_streets` + `core.core_street_names`

### Direct Copies

- `canonical_name` -> `core.core_streets.canonical_name`
- `geom` -> `core.core_streets.geom`

### Normalized Or Transformed

- `road_class_id` is not currently stored on `core.core_streets` in this schema and therefore remains part of lineage or future enrichment logic rather than a direct destination field.
- `source_snapshot_id`, `external_id`, and `source_refs` are preserved as lineage context, not copied directly into the street row.

### Derived Values

- `public_id` is generated by core.
- `admin_area_id` should be assigned where reliable reviewed area mapping exists.
- `source_type_id` is derived from lineage.
- `is_active`, `created_at`, and `updated_at` are assigned by core defaults and workflow outcome.

### Names

- The approved canonical name should create the primary row in `core.core_street_names`.
- Additional reviewed names, when available, should be inserted as alternate rows.

### Duplicates

- If the candidate matches an existing core street geometry/name identity, update instead of insert.
- Avoid creating duplicate street records for the same real-world road segment when review has already matched it.

## 4. `staging.staging_bus_stop_candidates` -> `core.core_bus_stops` + `core.core_bus_stop_names`

### Direct Copies

- `canonical_name` -> `core.core_bus_stops.name`
- `point_geom` -> `core.core_bus_stops.geom`

### Normalized Or Transformed

- `name_local` should be derived only when reviewed name data or approved localized content exists.
- `source_snapshot_id`, `external_id`, and `source_refs` stay in lineage records, not in the core stop row.

### Derived Values

- `public_id` is generated by core.
- `admin_area_id` is assigned from reviewed admin area linkage where available.
- `source_type_id` is derived from lineage.
- `stop_code` should be filled only if a reviewed stable code exists.
- `is_active` is assigned from workflow state.

### Names

- Insert the approved primary display name into `core.core_bus_stop_names`.
- Add additional localized or alternate names if review provides them.

### Duplicates

- If a candidate clearly matches an existing stop, update that stop instead of inserting a new one.
- Stop matching should prioritize reviewed identity and location quality over raw candidate confidence alone.

## 5. `staging.staging_bus_route_candidates` -> `core.core_bus_routes` + `core.core_bus_route_variants` + `core.core_bus_route_names`

### Direct Copies

- `route_code` -> `core.core_bus_routes.route_code`
- `public_name` -> `core.core_bus_routes.public_name`
- `geom` -> `core.core_bus_route_variants.geom`

### Normalized Or Transformed

- The candidate represents both route identity and an initial geometry variant.
- A route-level row should be created or updated in `core.core_bus_routes`.
- A variant row should then be created in `core.core_bus_route_variants` for the approved geometry.
- `external_id`, `source_snapshot_id`, and `source_refs` remain part of lineage context rather than route identity columns.

### Derived Values

- `source_type_id` is derived from candidate lineage.
- `variant_code` should be derived from approved business rules for route variant identity.
- `distance_m` may be derived from geometry if needed, or left null initially.
- `operator_name`, `route_type`, and `directionality` should be populated only when approved data exists.

### Names

- Insert the approved public name into `core.core_bus_route_names` as the primary route name.
- Additional localized or alternate route names should be added if reviewed source content supports them.

### Publish And Duplicates

- Route approval should result in core insert or update followed by publish workflow records.
- If the route already exists, update the route and create or update the appropriate variant instead of duplicating route identity.
- Duplicate handling should distinguish between:
  - same route identity with new geometry variant
  - same route identity with corrected name
  - genuinely separate routes that only look similar

## Practical Promotion Sequence

- resolve review outcome
- determine whether the staging candidate matches an existing core entity
- insert or update the core entity
- assign admin area linkage where reliable
- insert name rows
- insert source lineage rows where defined
- create initial or next version row where defined
- set publish status according to workflow state
- create publish item in the system workflow
